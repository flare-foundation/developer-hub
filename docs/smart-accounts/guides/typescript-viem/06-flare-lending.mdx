The diagram bellow showcases an example Flare lending workflow.
A user requests a lone from the lending contract, sending FXRP as backing.
They are issued a lending token (for example, USDT0), which they bridge back to XRPL.
In this guide, we are going to look at how this workflow can be performed entirely through smart accounts' custom instructions.

## Workflow

Let us begin by taking a look at the Flare lending workflow in more detail.

1. XRPL user sends a custom instruction for depositing collateral and taking a loan.
   The instruction includes the following five custom calls:
   1. Approve the transfer of the collateral amount of FXRP from the XRPL user's personal account to the lending contract.
   2. Deposit the collateral to the lending protocol contract.
   3. Take a loan.
   4. Approve the transfer of the USDT0 to the bridge contract.
   5. Initiate bridge of USDT0 to XRPL.
2. The operator server bridges the instruction to the Flare chain, using the Flare data connector (FDC).
3. The `MasterAccountController` contract delegates the custom instruction to the XRPL user's `PersonalAccount` contract.
4. XRPL user's `PersonalAccount` deposits collateral to the lending protocol contract and takes a loan.
5. The lending protocol transfers USDT0 to the XRPL user's `PersonalAccount`.
6. XRPL user's `PersonalAccount` deposits USDT0 to the bridge contract.
7. The USDT0 core vault on XRPL transfers USDT0 to the XRPL user's account.

:::info
In the following example, a single script is responsible for performing the entire process.
In an actual build:
- an XRPL wallet would prepare and send the custom instruction for the user and
- a separate service would bridge USDT0 to XRPL.

The latter could be a server run by an entity or, more securely, a TEE extension.
:::
